# POC-001 — Nextcloud Files Platform Validation

**Status:** Closed — candidate approved for a future implementation sprint

**Closed On:** 2026-07-21

**Branch:** `nextcloud`

**Classification:** Platform Enhancement

**Related Sprint:** Sprint 005 — Collaboration Platform

**Related Spike:** SPIKE-001 — Collaboration Architecture Evaluation

---

# Objective

Validate Nextcloud as a focused file synchronization and sharing service for
HomeLab07 by reusing the platform capabilities and engineering patterns proven
during Sprint 005.

The PoC must determine whether Nextcloud Files provides a more maintainable
administrative and customization model without weakening direct NAS file
recoverability.

The PoC is a clean deployment. It does not migrate OwnCloud data, users,
shares, configuration, or metadata.

OwnCloud is stopped for the duration of the test. Nextcloud becomes the only
active collaboration service in the HomeLab07 lifecycle and publication path.
The existing OwnCloud database, NAS data, private configuration, and repository
files remain intact for rollback.

---

# Decision Context

Sprint 005 demonstrated that HomeLab07 can host a collaboration service using
shared platform capabilities:

- MariaDB;
- Valkey;
- Nginx Proxy Manager;
- Cloudflare Dynamic DNS;
- shared Docker networks;
- NAS-backed persistence;
- the HomeLab07 operation layer;
- private environment configuration.

Nextcloud consumes the same capabilities. The PoC must reuse them without
deploying application-specific replacements.

This PoC is not a new platform design. It is a controlled replacement
candidate evaluation using an established service onboarding pattern.

---

# Hypothesis

HomeLab07 can deploy a minimal Nextcloud Files service by reusing the Sprint
005 architecture while improving:

- administrative diagnostics;
- filesystem reconciliation;
- declarative branding;
- application-profile control;
- operational documentation.

The hypothesis is valid only if Nextcloud preserves the HomeLab07 storage,
security, reproducibility, and recovery principles.

# Approved PoC Architecture

```text
Internet
    │
    ▼
Cloudflare Dynamic DNS
    │
    ▼
Nginx Proxy Manager
    │
    ▼
Nextcloud Files
    ├── Shared MariaDB
    ├── Shared Valkey
    ├── Dedicated NAS-backed persistent root
    └── Dedicated cron service
```

OwnCloud remains installed but stopped during the PoC. Its persistent state is
preserved and must not be mounted, modified, deleted, or reused by Nextcloud.

Nextcloud uses:

- a separate service name;
- a separate container name;
- a separate database and database user;
- a separate private environment file;
- a separate NAS share;
- the existing collaboration publication slot, redirected to Nextcloud for the
  test;
- no existing OwnCloud runtime state.

Only one collaboration application is active at a time:

```text
PoC active:     OwnCloud stopped  → Nextcloud active
PoC rollback:   Nextcloud stopped → OwnCloud active
```

# Database Provisioning Design

## Objective

Adapt the database provisioning pattern proven in Sprint 005 for Nextcloud
without modifying the existing OwnCloud scripts.

## Required Behavior

The Nextcloud database operations must:

- read application values from the selected private environment file;
- validate database, username, charset, and collation identifiers;
- safely quote password values;
- create the database idempotently;
- create or update the application database user;
- grant privileges only on the application database;
- display sanitized results;
- avoid printing credentials;
- require exact interactive confirmation before dropping state;
- reject missing, empty, or malformed values.

## PoC Private Values

The private file is expected at:

```text
HomeLab07.private/env/nextcloud.env
```

The repository example must use placeholders only:

```dotenv
NEXTCLOUD_DB_HOST=homelab07-mariadb
NEXTCLOUD_DB_NAME=nextcloud
NEXTCLOUD_DB_USERNAME=nextcloud
NEXTCLOUD_DB_PASSWORD=replace-with-a-strong-password
NEXTCLOUD_DB_CHARSET=utf8mb4
NEXTCLOUD_DB_COLLATION=replace-with-validated-nextcloud-collation
```

The exact collation must be validated against the pinned Nextcloud version and
the existing MariaDB 11.4 configuration before provisioning.

## Isolation And Preservation Requirement

Existing OwnCloud commands remain unchanged and available for rollback:

```text
operation/owncloud-db-create.sh
operation/owncloud-db-drop.sh
```

The new commands are independent:

```text
operation/nextcloud-db-create.sh
operation/nextcloud-db-drop.sh
```

Each script must read only its corresponding private environment file. A
Nextcloud database command must never accept or modify OwnCloud database
identifiers.

# Persistent Storage

## Dedicated NAS Share

Create a new Rockstor share. Recommended logical name:

```text
homelab07-nextcloud
```

The real host path belongs only in:

```text
HomeLab07.private/env/nextcloud.env
```

Example placeholder:

```dotenv
NEXTCLOUD_ROOT=/path/to/homelab07-nextcloud
```

## Candidate Layout

```text
homelab07-nextcloud/
├── html/
└── data/
```

Candidate mounts:

```text
${NEXTCLOUD_ROOT}/html -> /var/www/html
${NEXTCLOUD_ROOT}/data -> /var/www/html/data
```

The final mount layout must be validated before implementation. The official
image uses `/var/www/html` for installation state and updates, while allowing
separate mounts for data, configuration, custom applications, and themes.

The PoC should prefer the smallest layout that:

- survives container recreation;
- keeps user data directly identifiable;
- supports image upgrades;
- has unambiguous backup boundaries;
- avoids unnecessary bind mounts;
- remains understandable from the NAS.

## Storage Ownership

- Nextcloud is the normal writer to its internal data directory.
- The NAS remains the persistent storage platform.
- Jellyfin integration is not part of this PoC.
- Administrative filesystem writes are not part of normal operation.
- External file changes must be followed by a documented `occ files:scan`
  procedure.
- Runtime UID/GID and resulting NAS permissions must be documented from actual
  evidence.

## Encryption Policy

Nextcloud server-side encryption must remain disabled.

Validation must include:

```bash
docker exec --user www-data homelab07-nextcloud \
  php occ encryption:status
```

Expected result:

```text
enabled: false
```

Uploaded files must remain readable directly from the NAS after Nextcloud is
stopped.

# Background Jobs

Nextcloud requires reliable background processing. AJAX-triggered jobs are not
accepted for the PoC.

The candidate Compose design uses a second service based on the same pinned
image:

```text
homelab07-nextcloud
homelab07-nextcloud-cron
```

The cron service:

- runs `/cron.sh`;
- shares the required Nextcloud persistent volumes;
- uses the same private application configuration;
- has no published host ports;
- does not attach to `homelab07-proxy`;
- starts only after required platform dependencies are available.

Nextcloud must be configured for cron mode:

```bash
docker exec --user www-data homelab07-nextcloud \
  php occ background:cron
```

The PoC must verify the last background-job execution time.

# Valkey Integration

The shared Valkey service is reused as a Redis-compatible backend for:

- distributed cache;
- transactional file locking.

APCu remains the local cache.

Candidate private configuration:

```dotenv
REDIS_HOST=homelab07-valkey
REDIS_HOST_PORT=6379
```

The exact image variables must be validated before implementation.

The PoC must verify:

- `memcache.local` uses APCu;
- `memcache.distributed` uses Redis-compatible storage;
- `memcache.locking` uses Redis-compatible storage;
- Nextcloud reaches `homelab07-valkey` only through the internal network;
- cache keys do not conflict with another application consumer.

No dedicated Valkey instance is approved.


The existing collaboration public hostname is environment-specific and must

Candidate private settings include:

```dotenv
NEXTCLOUD_TRUSTED_DOMAINS=nextcloud.example.com

OVERWRITEPROTOCOL=https
```

network and generated `config.php`.

The Nginx Proxy Manager procedure must use placeholders for:

- public hostname;
- forward host;
- forward port;
- certificate selection.

Forward host and port:


# Operation Layer Integration

The PoC must preserve the operation layer as the public lifecycle interface.

Expected service command:

```bash
./operation/compose.sh nextcloud <compose-command>
```

Expected lifecycle integration:

- `start.sh` starts Nextcloud after MariaDB, Valkey, and Nginx Proxy Manager;
- `stop.sh` stops Nextcloud before shared dependencies;
- `status.sh` reports both application and cron service state;
- the storage check remains non-destructive;
- database creation remains explicit and auditable.

During the PoC, the lifecycle scripts must start and report Nextcloud instead
of OwnCloud. They must not start both collaboration services.

The OwnCloud service definition and service-specific operation scripts remain
in the repository for rollback. Rollback must restore the lifecycle selection,
redirect the existing Proxy Host to OwnCloud, stop Nextcloud, and then start
OwnCloud.

# Implementation Sequence

## Phase 1 — Platform Reuse Preparation

1. Confirm the pinned Nextcloud image and supported MariaDB version.
2. Create the dedicated Rockstor share.
3. Create the private `nextcloud.env` from the repository example.
4. Adapt the proven database scripts as Nextcloud-specific create and drop
   operations.
5. Confirm that the existing OwnCloud database scripts and state remain
   unchanged.
6. Stop OwnCloud through the HomeLab07 operation layer.

## Phase 2 — Isolated Deployment

1. Create `services/nextcloud/`.
2. Define application and cron services.
3. Attach existing networks.
4. Provision the independent database.
5. Validate the NAS path.

1. Complete non-interactive first-run initialization.
2. Do not install recommended Hub applications.
3. Inventory enabled applications.
4. Disable approved optional applications.
5. Configure cron mode.
6. Configure APCu and shared Valkey.

## Phase 4 — Publication And Branding

1. Redirect the existing collaboration Proxy Host to Nextcloud.
2. Validate HTTPS and trusted proxies.
3. Apply HomeLab07 branding through supported commands.
4. Recreate the container and verify persistence.

## Phase 5 — Recovery Validation

1. Upload representative test files.
2. Stop Nextcloud.
3. Confirm direct NAS readability and checksums.
4. Add one controlled test file outside Nextcloud.
5. Start Nextcloud and run a targeted `files:scan`.
6. Verify the imported file.
7. Back up configuration, data, and database.
8. Perform a clean restore using only PoC data.

# Backup And Restore Boundary

A complete Nextcloud backup includes at minimum:

- the persistent Nextcloud root;
- the user data directory;
- generated Nextcloud configuration;
- installed custom applications, if any;
- custom theme assets, if any;
- a consistent dump of the Nextcloud database;
- the pinned deployment definition;
- required private configuration through the approved private backup process.

Backing up only user files preserves content but does not preserve a complete
Nextcloud instance, shares, users, application state, or file metadata.

The PoC restore must:

1. stop application writes;
2. restore filesystem state and database from a consistent recovery point;
3. restore the same pinned application version;
4. validate configuration and permissions;
5. run documented post-restore commands;
6. verify file checksums and application state;
7. treat an application upgrade as a separate operation.

Backup automation remains outside this PoC and belongs to Sprint 010.

# Risks

| Risk | Impact | Mitigation |
|---|---|---|
| Generated configuration differs from declared environment | Recreated containers may retain stale values | Inspect `config.php`, document precedence, and validate recreation |
| `/var/www/html` persistence is misunderstood | Upgrade or restore failure | Validate official image lifecycle before accepting mount layout |
| Shared Valkey consumers interfere | Locking or cache anomalies | Validate key isolation and observe shared service activity |
| Cron does not share the correct state | Background jobs fail or run against stale code | Use the same pinned image and validated shared mounts |
| Optional Hub applications are installed during first run | Larger attack and maintenance surface | Disable recommended-app installation and validate enabled inventory |
| A Nextcloud DB script targets OwnCloud state | Existing service regression | Preserve OwnCloud scripts unchanged and validate strict environment and identifier isolation |
| Direct filesystem changes bypass Nextcloud metadata | Stale file index | Restrict external writes and document targeted `files:scan` |
| Fast Nextcloud major-release cadence increases maintenance | More frequent upgrade validation | Pin a supported production release and separate upgrade sprints |
| Branding commands store runtime state only | Recreation loses customization | Automate supported commands and validate recreation |
| Existing collaboration endpoint receives PoC traffic | Users may treat test data as permanent | Announce the test window and use synthetic or disposable data only |
| Rollback accidentally modifies OwnCloud state | Existing test state could be lost | Keep OwnCloud stopped and preserve its database, NAS share, environment, and service files |
| Lifecycle scripts start both applications | Ambiguous ownership and unnecessary resource use | Enforce one active collaboration service and validate container state |

# Acceptance Criteria

The PoC is successful only when:

- Nextcloud deploys through the HomeLab07 operation layer;
- OwnCloud remains stopped and its persistent state remains unchanged;
- Nextcloud is the only active collaboration service;
- MariaDB and Valkey are reused without dedicated replacements;
- Nextcloud uses an independent database, user, storage root, and runtime
  state;
- no host ports are published;
- the existing collaboration Proxy Host routes only to Nextcloud during the
  test;
- only the approved Files-focused application profile is active;
- cron executes reliably;
- transactional locking uses shared Valkey;
- server-side encryption is disabled;
- uploaded files are readable directly from the NAS;
- controlled external file addition is reconciled through `files:scan`;
- branding is applied through supported commands and survives recreation;
- configuration and data survive container recreation;
- a clean backup and restore succeeds with test data;
- documentation contains no secrets or environment-specific values;
- all applicable static, shell, runtime, and security validation passes.

The PoC does not approve a production migration. Its result must produce a
separate evidence-based decision record.

Rollback must also be demonstrated: stop Nextcloud, restore the Proxy Host and
lifecycle selection for OwnCloud, start OwnCloud, and confirm that its preserved
state remains accessible.

# Expected Decision Record

After implementation, HomeLab07 must answer:

> Does a minimal Nextcloud Files deployment provide enough administrative,
> recovery, and customization benefit to justify replacing the already stable
> OwnCloud service in a future sprint?

Valid outcomes are:

- approve a future Nextcloud implementation sprint;
- continue with OwnCloud;
- extend the PoC for unresolved recovery evidence;

Preference, feature count, and visual appearance alone are not sufficient for

# Closure Decision

## Outcome

POC-001 is closed with the following decision:

> Approve a future Nextcloud implementation sprint, while withholding approval
> for production migration until recovery, rollback and data-integrity gates
> are demonstrated.

This outcome accepts Nextcloud as the preferred collaboration-platform
candidate for continued HomeLab07 engineering. It does not decommission
OwnCloud, authorize data migration or classify all PoC acceptance criteria as
complete.

## Evidence Obtained

- The pinned application and cron definitions resolve through Docker Compose.
- Nextcloud deployed through the HomeLab07 operation layer.
- The web container reached normal Apache operation and returned HTTP 200 from
  `/status.php` health checks.
- The application UI became reachable through Nginx Proxy Manager at the
  environment-specific HTTPS endpoint.
- Shared MariaDB and Valkey were reused without dedicated replacements.
- Nextcloud used its independent database configuration and dedicated NAS
  root.
- The NAS data directory was owned by runtime UID/GID `33:33`.
- No Nextcloud host port was published by the Compose definition.
- The first-run `NEXTCLOUD_INIT_HTACCESS` restart loop was diagnosed, removed
  and documented.
- `trusted_proxies` was corrected from an invalid Docker hostname to a private
  proxy-network CIDR supplied outside Git.
- Static Compose, shell syntax and repository diff validation succeeded.

## Gates Carried Into The Future Sprint

The following were not demonstrated during the closing session and remain
mandatory before production migration:

- record the exact image digest and target architecture;
- validate the Files-only enabled application inventory;
- confirm cron execution and generated APCu/Valkey configuration;
- confirm server-side encryption remains disabled;
- verify uploaded-file checksums directly from NAS storage;
- validate a targeted external-file reconciliation with `files:scan`;
- apply branding and prove it survives container recreation;
- perform a consistent backup and clean restore with disposable test data;
- demonstrate rollback to preserved OwnCloud state;
- capture final `occ setupchecks` and application-log results.

Until those gates pass, Nextcloud remains a successful deployment candidate,
not the approved production replacement for OwnCloud.
